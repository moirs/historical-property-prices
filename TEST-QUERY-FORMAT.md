# HM Land Registry SPARQL Query Format (Corrected)

## Sample Query

The following is an example of a SPARQL query generated by the corrected query builder using the proper HM Land Registry ontologies:

```sparql
PREFIX rdf: <http://www.w3.org/1999/02/22-rdf-syntax-ns#>
PREFIX rdfs: <http://www.w3.org/2000/01/rdf-schema#>
PREFIX owl: <http://www.w3.org/2002/07/owl#>
PREFIX xsd: <http://www.w3.org/2001/XMLSchema#>
PREFIX sr: <http://data.ordnancesurvey.co.uk/ontology/spatialrelations/>
PREFIX ukhpi: <http://landregistry.data.gov.uk/def/ukhpi/>
PREFIX lrppi: <http://landregistry.data.gov.uk/def/ppi/>
PREFIX skos: <http://www.w3.org/2004/02/skos/core#>
PREFIX lrcommon: <http://landregistry.data.gov.uk/def/common/>

SELECT ?paon ?saon ?street ?town ?county ?postcode ?amount ?date ?category
WHERE {
  VALUES ?postcode {"PL68RU"^^xsd:string}
  ?addr lrcommon:postcode ?postcode .
  ?transx lrppi:propertyAddress ?addr ;
          lrppi:pricePaid ?amount ;
          lrppi:transactionDate ?date ;
          lrppi:transactionCategory/skos:prefLabel ?category .
  OPTIONAL {?addr lrcommon:county ?county}
  OPTIONAL {?addr lrcommon:paon ?paon}
  OPTIONAL {?addr lrcommon:saon ?saon}
  OPTIONAL {?addr lrcommon:street ?street}
  OPTIONAL {?addr lrcommon:town ?town}
  FILTER(?date >= "2022-01-01"^^xsd:date)
  FILTER(?date <= "2023-12-31"^^xsd:date)
}
ORDER BY ?amount
LIMIT 50
```

## Key Changes Made

### 1. **Correct Prefixes**
The query now uses the proper HM Land Registry ontologies:
- `lrcommon:` - Land Registry Common properties (postcode, paon, saon, street, town, county)
- `lrppi:` - Land Registry Price Paid Data properties (propertyAddress, pricePaid, transactionDate, transactionCategory)
- `skos:` - SKOS vocabulary for category labels (prefLabel)
- Standard RDF/RDFS/OWL/XSD prefixes

**Previously (INCORRECT):**
```sparql
PREFIX ppd: <http://purl.org/voc/ppd#>  # Wrong - PPD doesn't match endpoint
```

### 2. **VALUES Clause**
Uses SPARQL 1.1 VALUES syntax for postcode parameter:
```sparql
VALUES ?postcode {"PL68RU"^^xsd:string}
```

**Previously (INCORRECT):**
```sparql
FILTER(?postcode = "PL68RU")  # Wrong - postcode comes from address
```

### 3. **Address Pattern**
Correctly models the HM Land Registry data structure:
```sparql
?addr lrcommon:postcode ?postcode .
?transx lrppi:propertyAddress ?addr ;
        lrppi:pricePaid ?amount ;
        lrppi:transactionDate ?date ;
        lrppi:transactionCategory/skos:prefLabel ?category .
```

**Previously (INCORRECT):**
```sparql
?transaction ppd:propertyAddress ?address ;
             ppd:postcode ?postcode ;
             ppd:pricePaid ?price ;
             ppd:transactionDate ?date .
```

### 4. **SELECT Variables**
Returns address components separately for proper reconstruction:
```sparql
SELECT ?paon ?saon ?street ?town ?county ?postcode ?amount ?date ?category
```

- `paon`: Property Address Object Number (house/building number)
- `saon`: Secondary Address Object Number (apartment/flat number)
- `street`: Street name
- `town`: Town/city
- `county`: County
- `postcode`: Full postcode
- `amount`: Sale price in GBP
- `date`: Transaction date (ISO format)
- `category`: Transaction category (e.g., "Standard Price Paid Entry")

**Previously (INCORRECT):**
```sparql
SELECT DISTINCT ?address ?postcode ?price ?date
```

### 5. **OPTIONAL Address Properties**
Uses OPTIONAL blocks for properties that may not exist:
```sparql
OPTIONAL {?addr lrcommon:county ?county}
OPTIONAL {?addr lrcommon:paon ?paon}
OPTIONAL {?addr lrcommon:saon ?saon}
OPTIONAL {?addr lrcommon:street ?street}
OPTIONAL {?addr lrcommon:town ?town}
```

### 6. **Price Filter**
Uses the correct property name and variable:
- `?amount` instead of `?price`
- `lrppi:pricePaid` property (implicit in the WHERE clause above)

### 7. **ORDER BY and LIMIT**
Maintains proper query structure:
```sparql
ORDER BY ?amount
LIMIT 50
```
